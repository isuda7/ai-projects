# 📚 가이드 시스템 설계 및 아키텍처 컨텍스트 (Guide System Architecture)

이 문서는 프로젝트의 가이드 영역(`/guide`) 및 퍼블리싱 대시보드에 대한 **구조적/시각적 설계 기준**을 정의합니다. AI가 가이드 화면이나 컴포넌트를 작업할 때 임의로 레이아웃을 생성하지 않도록 돕는 핵심 지침서입니다.

## 1. 📝 화면 생성 원칙 (MD 프롬프트 기반)
가이드 및 대시보드 내의 어떠한 화면도 AI가 임의의 형태로 구성해서는 안 됩니다.
반드시 사전에 정의된 **유형별 프롬프트 가이드 문서(.md)**를 통해 마크업과 컴포넌트 구조를 생성해야 합니다.
*   대시보드 화면 생성/수정: `PROMPT_DASHBOARD.md` 참조
*   신규 UI 컴포넌트 가이드 생성: `PROMPT_GUIDE_01_BASE.md` (표준 양식) 참조
*   가이드 유형별 생성/수정: `PROMPT_GUIDE_*.md` (_02 번 부터 파일 순서대로 메뉴 추가) 참조

## 2. 📂 가이드와 사용자(Front) 영역의 폴더/라우팅 분리
*   **사용자 화면**: `src/pages/beforeLogin/` (로그인 전), `src/pages/afterLogin/` (로그인 후) 또는 일반 `src/pages/` 하위에 위치하며, 전역 레이아웃인 `Layout` 컴포넌트(`src/components/layout/layout.tsx`)를 사용합니다.
*   **가이드 화면**: 모든 문서는 `src/pages/guide/` 폴더 하위에 위치해야 하며, 반드시 `guide-layout.tsx`로 감싸져(Wrapping) 독립된 도메인으로 작동해야 합니다.

## 3. 🧩 공통 컴포넌트 정책 (Header / Hooter / Layout)
*   사용자 화면과 가이드 화면은 모든 레이아웃을 공통으로 사용하지 않는다. (완벽하게 분리되어야 한다.)
*   가이드 내에서의 프리뷰로 보여주고자 하는 사용자 화면의 컴포넌트만 가져올 수 있다.
*   헤더 내부에는 현재 접속 위치를 직관적으로 알 수 있도록 포인트 컬러(Cyan: `#00adc0`)와 하단 인디케이터(밑줄)를 활용한 Active 라우팅 강조(`top_nav_link_active`) 처리가 되어 있어야 합니다.

## 4. 📐 가이드 내부 레이아웃의 이원화 정책
가이드 레이아웃(`guide-layout.tsx`)은 내부적으로 라우팅 경로(`useLocation`)를 감지하여 2가지 모드로 레이아웃을 다르게 렌더링해야 합니다. 

### A. 퍼블리싱 대시보드 모드 (`/guide/dashboard`)
*   **목적**: 화면 개발 진척도, 경로, URL을 표 형태로 한눈에 밀도 있게 관리하는 화면.
*   **레이아웃 특징**: 테이블에 많은 데이터가 들어가야 하므로 **좌측 사이드바 메뉴를 숨기고(Unmount)**, 본문 컨테이너가 브라우저 **전체 너비(`max-width: 100%`)**를 시원하게 차지하도록 `dashboard_inner` 클래스를 부여합니다.

### B. 컴포넌트/문서 가이드 모드 (대시보드 외 모든 `/guide/*` 경로)
*   **목적**: 버튼, 컬러, 타이포그래피 등 디자인 시스템의 사용법을 열람하는 문서 화면.
*   **레이아웃 특징**: 가독성이 최우선이므로 **좌측에 사이드바(Guide Menu)를 항상 노출**하고, 우측 본문 영역은 화면을 꽉 채우지 않고 **가운데 정렬된 제한된 너비(`max-width: 1024px`)**를 가지도록 `content_inner` 클래스를 부여합니다.

## 5. 🎨 디자인 컨셉 지향점
모든 가이드 및 사용자 화면은 화려함보다는 정보 전달력이 뛰어난 **'Professional Modern'** 스타일을 지향합니다.
*   **여백과 단위**: 모든 단위는 `rem` 대신 직관적인 **`px` 단위로 통일**하여 소통 오류를 줄입니다.
*   **프롬프트 기록 자동화**: `usePromptStore`를 통해 매 작업 단계의 프롬프트와 의사결정 과정을 대시보드에서 조회할 수 있도록 설계합니다.
