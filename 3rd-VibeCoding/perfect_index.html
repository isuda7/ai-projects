작업의 규모와 성격에 따라 두 가지 개발 방식을 선택할 수 있습니다.

아이디어에서 완성까지, 거침없는 3단계 전개

1️⃣

큰 그림 던지기 (초안)

완벽한 기획서 없이, 만들고자 하는 서비스의 핵심 컨셉과 뼈대만 AI에게 묘사하여 빠르게 첫 결과물을 뽑아냅니다.

2️⃣

내용 및 기능 채우기

생성된 초안을 바탕으로 더미 데이터를 넣거나, 화면 간의 이동 로직 등 실질적인 살을 붙입니다.

3️⃣

디테일 개선 (폴리싱)

애니메이션, 모바일 반응형 대응, 예외 처리 등 완성도를 끌어올리기 위한 세부 디테일을 다듬습니다.

실무 단계에서 무한 반복되는 6가지 기본 공식

어떤 기능을 만들든 이 6가지 순서만 똑같이 반복하면 됩니다.

1️⃣

요구사항 구체화

어떤 기능을 만들지 명확한 입력(Input)과 기대 출력(Output)을 정의합니다.

2️⃣

뼈대(Skeleton) 구축

세부 로직을 작성하기 전 라우팅, 폴더 구조, 데이터 모델 등 아키텍처를 우선 세팅합니다.

프롬프트 예시 보기 ↗
3️⃣

마이크로 태스크 진행

UI 마크업, API 연동 등 하나의 작업만 독립적으로 AI에게 지시하고 검증합니다.

프롬프트 예시 보기 ↗
4️⃣

에러 복붙 디버깅

오류 발생 시 에러 로그 전문과 상황을 그대로 복사해 AI에게 해결책을 묻습니다.

프롬프트 예시 보기 ↗
5️⃣

코드 리뷰 및 리팩토링

코드가 작동하더라도 "더 나은 패턴이나 취약점은 없는지" 되물어 코드를 다듬습니다.

6️⃣

컨텍스트 업데이트

새로운 라이브러 추가나 요구사항 변경 시, AI의 메모리(Context)를 다시 갱신해줍니다.

실무 단계 가이드 및 작성 기법

명확한 기획이 있을 때와 없을 때의 접근법, 그리고 필수 바이브코딩 작성 기법입니다.

상황 A

프롬프트가 있는 경우 (명확한 기획)

기획안이 확고하다면 '구체적 지시와 제한'이 핵심입니다. 기대하는 출력값(Output)을 명시하고, 사용할 프레임워크나 라이브러리를 엄격히 고정하여 AI가 딴 길로 새지 않도록 통제합니다.

상황 B

프롬프트가 없는 경우 (아이디어 단계)

스펙이 불명확하다면 '탐색적 대화'로 시작합니다. "이런 서비스를 만들 건데 어떤 아키텍처가 좋을까?"라고 AI의 제안을 먼저 이끌어낸 뒤, 초안을 바탕으로 점차 요구사항을 구체화(Refining)합니다.

💡 바이브코딩 작성기법 기본 가이드

  • ✓ 'How'보다 'What': 로직을 어떻게 짤지 장황하게 지시하기보다, 달성할 최종 목표와 결과물 형태를 명확히 지시하세요.
  • ✓ 에러 로그의 활용: 오류 발생 시 당황하지 말고 터미널의 에러 로그 전문과 발생 상황을 복사해서 그대로 붙여넣어 원인을 분석시키세요.
  • ✓ 마이크로 태스크 분할: 한 번에 거대한 기능 구현을 요구하면 환각(Hallucination)이 발생합니다. 최소 단위로 쪼개어 하나씩 해결하세요.
  • ✓ 비판적 코드 리뷰: AI가 작성한 코드를 무비판적으로 수용하지 말고 "잠재적인 버그나 성능 이슈, 보안 취약점은 없는지 검토해줘"라고 재요청하세요.
