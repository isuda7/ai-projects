VibeCoding Process

바이브 코딩의 절차적인 접근 방식과 워크플로우를 안내합니다.

⚙️
환경설정

Phase 1. 전체 방향 잡기

  • • 프로젝트 기술 스택(언어/프레임워크) 정의
  • • 폴더 구조 및 아키텍처 규칙 세팅
  • • 코딩 컨벤션 및 네이밍 규칙 확립
  • 🔥 핵심: AI가 지켜야 할 절대 규칙(System Prompt) 주입
시스템 프롬프트 템플릿 보기 ↗
📐
가이드

Phase 2. 뼈대 및 디자인

  • • 프론트 페이지 목록(대시보드) 정리
  • • 파운데이션(디자인 토큰 등) 세팅
  • • 공통 UI 컴포넌트 산출물 정리
  • 🔥 핵심: 본격적인 기능구현 전 가이드 정리를 위한 단계
가이드 프롬프트 템플릿 보기 ↗
🛠️
기능구현

Phase 3. 핑퐁하며 구현하기

  • • 정해진 규칙 위에서 지속적인 대화(핑퐁) 진행
  • • 기능 검증 및 에러 발생 시 즉각 피드백
  • • 구현된 코드의 주기적인 리뷰 및 리팩토링
  • 🔥 핵심: 거대한 지시 대신 작은 기능 단위(Chunk)로 쪼개서 요청

실무 단계에서 무한 반복되는 6가지 기본 공식

어떤 기능을 만들든 이 6가지 순서만 똑같이 반복하면 됩니다.

1

요구사항 구체화

어떤 기능을 만들지 명확한 입력(Input)과 기대 출력(Output)을 정의합니다.

2

설계 구축

세부 로직을 작성하기 전 라우팅, 폴더 구조, 데이터 모델 등 아키텍처를 우선 세팅합니다.

프롬프트 예시 보기 ↗
3

마이크로 태스크 진행

UI 마크업, API 연동 등 하나의 작업만 독립적으로 AI에게 지시하고 검증합니다.

프롬프트 예시 보기 ↗
4

에러 복붙 디버깅

오류 발생 시 에러 로그 전문과 상황을 그대로 복사해 AI에게 해결책을 묻습니다.

프롬프트 예시 보기 ↗
5

코드 리뷰 및 리팩토링

코드가 작동하더라도 "더 나은 패턴이나 취약점은 없는지" 되물어 코드를 다듬습니다.

프롬프트 예시 보기 ↗
6

컨텍스트 업데이트

새로운 라이브러 추가나 요구사항 변경 시, AI의 메모리(Context)를 다시 갱신해줍니다.

실무 단계 가이드 및 작성 기법

명확한 기획이 있을 때와 없을 때의 접근법, 그리고 필수 바이브코딩 작성 기법입니다.

접근 방식 A

결과 기반 프롬프트 도출 (역추론)

결과물을 직접 만들며 프롬프트를 역추론할 때는 '대화와 요약'을 활용합니다. 단편적인 기능들을 AI와 함께 구현하며 성공적인 결과를 찾아내고, "방금 우리가 만든 결과물을 처음부터 단번에 도출하려면 어떤 프롬프트가 필요할까?"라고 지시하여 AI 스스로 최적의 프롬프트를 역산하도록 유도합니다.

진행과정 템플릿 보기 ↗
접근 방식 B

프롬프트 자체 사전 제작 (설계)

프롬프트 자체를 먼저 제작할 때는 '구조화와 제약'이 핵심입니다. 프로젝트의 목적, 절대 어겨선 안 될 규칙, 기술 스택, 그리고 기대하는 출력 포맷(Output)을 마크다운 문법으로 뼈대부터 명확히 규격화하여, AI가 딴 길로 새지 않도록 완벽히 통제하는 지시문을 설계합니다.

💡 바이브코딩 작성기법 기본 가이드

  • ✓ 'How'보다 'What'
    구현 과정(How)을 CSS 속성으로 하나씩 지시하지 마세요.
    원하는 최종 결과물의 시각적 형태(What)만 명확하게 묘사하세요.
    예시 보기 ↗
  • ✓ 에러 로그의 활용
    레이아웃이 깨지거나 오류가 날 때 말로 추측해서 설명하지 마세요.
    문제 상황을 구체적으로 적고 에러 로그나 코드를 통째로 붙여넣으세요.
    예시 보기 ↗
  • ✓ 마이크로 태스크 분할
    거대한 기능 구현을 한 번에 요구하면 화면 누락이나 에러가 발생합니다.
    하나의 명확하고 작은 지시 단위로 쪼개어 하나씩 점진적으로 해결하세요.
    예시 보기 ↗
  • ✓ 비판적 코드 리뷰
    AI가 작성한 코드를 무비판적으로 수용하지 마세요.
    "잠재적인 버그나 성능 이슈, 보안 취약점은 없는지 시니어 관점에서 검토해줘"라고 재요청하세요.
