시스템 프롬프트

Overview

성공적인 바이브코딩을 위해 시스템 프롬프트 단계에서 준비해야 할 두 가지 핵심 축을 파악합니다.

🛠️

개발환경

▶ 기술 스택 (Tech Stack)

프로젝트에 사용될 프레임워크 및 핵심 라이브러리 종속성 지정

▶ 설정 파일 (Configurations)

package.json, tsconfig.json 등 프로젝트 베이스가 되는 설정 룰

▶ 코드 품질 (Lint & Format)

AI가 코드를 작성할 때 강제로 준수해야 할 ESLint, Prettier 컨벤션

▶ 환경 변수 (Env Variables)

.env 구조 및 시크릿 키, 포트 번호 등 로컬/운영 환경 관리 기준

🏗️

구조설계

▶ 디렉토리 구조

AI가 코드를 쉽게 탐색하고 수정할 수 있는 직관적인 폴더 트리 정의

▶ 아키텍처 패턴

컴포넌트 분리 원칙, 모듈화 등 시스템 전반의 설계 기준 수립

▶ 상태 및 라우팅

전역 상태 관리 전략과 화면 전환(라우팅) 계층 구조 설계

▶ IA 및 명세서

요구사항 정의서와 정보 구조도(IA) 작성을 통한 명확한 목표 설정

개발환경

프로젝트에 사용될 기술 스택, 설정 파일, 코드 품질 기준, 환경 변수 등 AI에게 주입할 핵심 베이스를 정의합니다.

1. 기술 스택 (Tech Stack)

어떤 언어와 프레임워크를 사용할지 명시합니다.

  • Core Framework: Next.js (React), Vue, Svelte, Vanilla JS
  • Language: TypeScript (Strict), JavaScript (ES6+)
  • Styling: Tailwind CSS, SCSS, Styled-components
  • UI Library / System: shadcn/ui, MUI, 디자인 토큰 자체 구축

2. 설정 파일 (Configurations)

프로젝트 루트의 핵심 설정 파일 기준을 세웁니다.

  • Package Manager: npm / yarn (버전 고정), pnpm, bun
  • TypeScript: tsconfig.json (Strict 모드, Alias)
  • Bundler & Config: next.config.js, vite.config.ts, webpack
  • Style Config: tailwind.config.ts, postcss.config.js, .editorconfig

3. 코드 품질 (Lint & Format)

AI가 코드를 작성할 때 지켜야 할 컨벤션입니다.

  • ESLint: Airbnb 룰 기반 (Custom strict), Standard
  • Prettier: Tab: 2, Single Quote, 세미콜론 강제
  • Naming Convention: 컴포넌트(Pascal), 변수(camel), BEM(CSS)
  • Publishing Rules: 시맨틱 태그 준수, WAI-ARIA 접근성 속성 포함

4. 자산 및 환경 변수 (Assets & Env)

정적 자산 관리 및 프론트엔드 환경 변수 기준입니다.

  • File 분리: .env.local, .env.production
  • Prefix Rule: NEXT_PUBLIC_, VITE_, REACT_APP_
  • 정적 자산(Assets): public/ 폴더 내 최적화 (WebP, SVG)
  • 보안 규칙: .gitignore에 .env 추가, API 키 노출 방지

구조설계

디렉토리 구조, 아키텍처 패턴, 상태 관리 및 라우팅 전략 등 시스템의 물리적/논리적 뼈대를 설계합니다.

1. 디렉토리 구조 (Directory)

AI가 프론트엔드 코드를 파악할 수 있는 폴더 트리입니다.

  • Component Structure: Feature-first (기능별 분리), Type-first
  • UI Component: Atomic Design (atoms, molecules), 공통 분리
  • Colocation: 컴포넌트와 스타일(CSS) 동일 폴더 내 배치
  • 정적 파일: public/images, public/icons 등 에셋 구조화

2. 아키텍처 패턴 (Architecture)

UI 렌더링과 뷰(View) 로직을 관리하는 패턴입니다.

  • Logic & UI 분리: Custom Hooks 패턴 (로직 추상화)
  • 컴포넌트 설계: Container / Presentational 패턴
  • 렌더링 최적화: Server Component와 Client 분리
  • 스타일링 패턴: Utility-first (Tailwind), CSS Modules

3. 상태 및 라우팅 (State/Routing)

앱의 화면 데이터 흐름과 전환 전략입니다.

  • Global State (UI): Zustand, Redux Toolkit, Jotai
  • Server State (API): React Query (TanStack), SWR
  • Routing: Next.js App Router (File-based), React Router
  • Data Fetching: SSR, SSG, CSR 렌더링 기준 정의

4. UI/UX 명세서 (UI & Spec)

퍼블리싱 전 화면 목표와 시각적 명세를 정의합니다.

  • Sitemap / Flow: 화면 전환도 및 레이아웃 구조 (GNB, LNB)
  • Component API: Props 인터페이스, Variant(상태) 정의
  • 상태 명세서: Hover, Active, Disabled 등 UI 상태 정의
  • 반응형 (Responsive): Mobile-first 기준 Breakpoint 명세

# VibeCoding System Prompt

## 1. 개발환경 (Development Environment)
- **Tech Stack:** Next.js 14 (App Router), React 18, TypeScript (Strict).
- **Styling:** Tailwind CSS, shadcn/ui.
- **Config & Lint:** ESLint (Airbnb-based strict rules), Prettier (tabWidth: 2, singleQuote: true).
- **Environment Variables:**
  - 사용 시 반드시 `process.env.NEXT_PUBLIC_` 접두사를 준수할 것.
  - 보안 정보는 클라이언트 사이드에 노출되지 않도록 서버사이드에서만 호출할 것.

## 2. 구조설계 (Structural Design)
- **Directory Structure (Feature-first):**
  - 도메인 단위로 폴더를 구성하며, `src/features/{domain}/` 내부에 components, hooks, api, types를 묶어서 배치할 것 (Colocation).
  - 공통 컴포넌트와 유틸리티는 `src/shared/`에 분리할 것.
- **Architecture Pattern:**
  - 비즈니스 로직과 UI를 엄격히 분리할 것. (Custom Hooks 패턴 권장)
  - 서버 데이터 페칭과 클라이언트 렌더링을 분리(Server Component vs Client Component)할 것.
- **State & Routing:**
  - **Global State:** Zustand를 사용하며 스토어는 도메인별로 작게 쪼개서 관리.
  - **Server State:** React Query (TanStack Query)를 사용하여 캐싱과 무효화를 관리.
  - **Routing:** Next.js App Router의 File-based 라우팅 규칙 준수 (layout.tsx, page.tsx).
- **Code Convention:**
  - 변수 및 함수는 camelCase, 컴포넌트는 PascalCase를 사용할 것.
  - 모든 함수의 인자와 반환값에 명시적인 TypeScript 타입을 지정할 것 (`any` 사용 금지).

당신은 최고의 시니어 프론트엔드 개발자입니다. 위 규칙을 어기지 말고 최적화되고 유지보수하기 쉬운 코드를 작성하십시오.
